# Review-Prompt: Portal-Install Robustheit (v0.8.0)

## Kontext

Das Portal-Install (`install-portal.yml`) ist **zweimal in Folge** in Phase 7 (Validierung) fehlgeschlagen:

1. **Fehler 1:** `POST /api/buy` → 500 "Kein Key-Praefix fuer Produkt: factory-gateway"
   - Ursache: Smoke-Test nutzte `factory-gateway` (Infrastruktur-Service ohne Lizenz-Prefix)
   - Fix: Smoke-Test auf `mitglieder-simple` umgestellt

2. **Fehler 2:** `POST /api/buy` → 500 "column license_hash of relation licenses does not exist"
   - Ursache: `portal-db` Ansible-Rolle fuehrte nur Migrationen bis v0.5.6 aus
   - Die SQL-Dateien `migrate-v062.sql` bis `migrate-v072.sql` existierten im Repo, waren aber **nie** in der Ansible-Rolle registriert
   - `license_hash` wird in `migrate-v070.sql` hinzugefuegt (ALTER TABLE)
   - Der Code (`license.js:createLicense`) schreibt `license_hash` seit v0.7.0 — aber die DB hatte die Spalte nie

## Grundprobleme

### 1. Keine Konsistenzpruefung zwischen SQL-Dateien und Ansible-Rolle

Neue Migrationen koennen als `.sql`-Datei erstellt werden, ohne dass jemand prueft ob sie auch in `portal-db/tasks/main.yml` registriert sind. Es gibt keinen Lint, keinen Test, keine CI-Pruefung dafuer.

### 2. Kein Schema-Versionscheck vor Smoke-Tests

Die Validierung (`portal-validate`) testet API-Endpunkte die bestimmte DB-Spalten voraussetzen, prueft aber nicht ob das Schema vollstaendig migriert ist. Wenn eine Migration fehlt, bekommt man einen kryptischen 500er statt einer klaren Fehlermeldung.

### 3. Manuelles Migrations-Registry (Copy-Paste-Anfaellig)

Jede Migration erfordert 3 identische Zeilen in `portal-db/tasks/main.yml`:
```yaml
- name: "vX.Y.Z Migration in Container kopieren"
  shell: docker cp ... portal-db:/tmp/migrate-vXYZ.sql
- name: "vX.Y.Z Migration ausfuehren"
  shell: docker exec portal-db psql -U portal -d portal -f /tmp/migrate-vXYZ.sql
- name: "vX.Y.Z Migration Ergebnis"
  debug: msg: ...
```
Das ist fehleranfaellig — vergisst man es, schlaegt das Install erst in Phase 7 fehl (nach ~10 Minuten).

### 4. Tarball-Timing-Problem

Wenn Fixes am Code gemacht werden waehrend ein Install laeuft, nutzt das laufende Install den alten Tarball. Es gibt keinen Mechanismus der warnt wenn der Tarball veraltet ist.

## Review-Aufgaben

### A. Migrations-Konsistenz sicherstellen

1. **Pruefe:** Stimmt die Reihenfolge der Migrationen in `portal-db/tasks/main.yml` mit den SQL-Dateien ueberein?
2. **Pruefe:** Sind alle `migrate-v*.sql` Dateien in `portal/src/db/` auch in der Ansible-Rolle registriert?
3. **Erstelle:** Ein Validierungsscript `scripts/validate-portal-migrations.sh` das automatisch prueft:
   - Jede `portal/src/db/migrate-v*.sql` Datei MUSS in `ansible/roles/portal-db/tasks/main.yml` referenziert sein
   - Keine Datei darf fehlen
   - Das Script soll in `pnpm lint:migrations` (neuer Script in root package.json) aufrufbar sein

### B. Dynamische Migrations-Ausfuehrung

Ersetze das manuelle Copy-Paste-Pattern durch eine Schleife:

```yaml
# Vorschlag: Alle migrate-v*.sql automatisch finden und ausfuehren
- name: "Migrations-Dateien finden"
  ansible.builtin.find:
    paths: "{{ portal_docker_compose_dir }}/src/db"
    patterns: "migrate-v*.sql"
  register: migration_files

- name: "Migrationen sortiert ausfuehren"
  ansible.builtin.shell: |
    docker cp {{ item.path }} portal-db:/tmp/{{ item.path | basename }}
    docker exec portal-db psql -U portal -d portal -f /tmp/{{ item.path | basename }}
  loop: "{{ migration_files.files | sort(attribute='path') }}"
  register: migration_results
```

**Review-Frage:** Ist dieser Ansatz sicher? Was passiert bei idempotenten vs. nicht-idempotenten Migrationen? Alle aktuellen Migrationen nutzen `IF NOT EXISTS` / `IF EXISTS` — ist das ausreichend?

### C. Schema-Validierung vor Smoke-Tests

Fuege einen Pre-Check in `portal-validate` ein, der vor den API-Tests prueft ob das Schema vollstaendig ist:

```yaml
- name: "Schema-Validierung: Pflicht-Tabellen und -Spalten pruefen"
  ansible.builtin.shell: |
    docker exec portal-db psql -U portal -d portal -tAq -c "
    SELECT column_name FROM information_schema.columns
    WHERE table_name = 'licenses' AND column_name = 'license_hash'"
  register: schema_check
  failed_when: schema_check.stdout | trim | length == 0
```

**Review-Frage:** Welche Spalten/Tabellen sind minimal noetig fuer die Smoke-Tests? Liste alle Abhaengigkeiten auf.

### D. Tarball-Konsistenz

Wie kann sichergestellt werden, dass der Tarball der beim Install verwendet wird, alle aktuellen Fixes enthaelt?

Optionen:
1. Hash-Check: `build-installer.sh` schreibt SHA256 in `dist/CHECKSUM`, `install.sh` vergleicht
2. Timestamp-Check: `install.sh` warnt wenn Tarball aelter als 1 Stunde
3. Git-Status-Check: `build-installer.sh` verweigert Build wenn uncommitted changes existieren

**Review-Frage:** Welche Option ist am praktischsten fuer den aktuellen Workflow (manuelles Install von lokalem Rechner)?

### E. Pflege-Regel

Ergaenze in `CLAUDE.md` die Pflege-Regeln:

> Bei jeder neuen Portal-Migration (`portal/src/db/migrate-v*.sql`) MUSS die Ansible-Rolle `portal-db/tasks/main.yml` aktualisiert werden. `pnpm lint:migrations` prueft dies automatisch.
