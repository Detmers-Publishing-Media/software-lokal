## Was v0.4.0 können MUSS (Akzeptanztest)

```
1. ansible-playbook install.yml → PROD steht, /ready gibt 200
2. ansible-playbook blue-green.yml → IAT hoch, Smoke grün, DNS geschwenkt,
   X-Served-By bestätigt neues PROD, alte Server NICHT gelöscht (Delay)
3. ansible-playbook nacht-stopp.yml --tags=stop → Server aus, Notification "stop ok"
4. ansible-playbook nacht-stopp.yml --tags=start → Server an, /ready 200, Notification "start ok"
5. ansible-playbook teardown.yml → Alles weg (nur tagged Server), Notification
6. Smoke-Test ROT → IAT abgerissen, PROD unberührt, Notification "abbruch"
7. Alle Secrets in Vault, alle Images version-pinned
8. Backup-Verifikation: dump size, repo count, git fsck → alles grün
```

**Der ultimative Akzeptanztest:**
1. `ansible-playbook install.yml` → PROD läuft, `/ready` grün, Notification erhalten
2. Ändere eine Infrastruktur-Datei (z.B. Poller-Update)
3. `ansible-playbook blue-green.yml` → IAT hoch, Smoke grün, DNS geschwenkt
4. `X-Served-By` bestätigt neues PROD, Backup verifiziert
5. Alte Server als `pending_delete` getaggt (nicht sofort gelöscht)
6. Gesamtzeit < 30 Minuten

## Architektur v0.4.0

### Server-Landschaft

```
PROD (8–22 Uhr solange nicht produktiv, danach 24/7)
├── Forgejo (Git + Actions)
├── Runner (act_runner, Host-systemd)
├── OpenClaw (Junior Agent)
├── Claude Code (Architect + Reviewer, in Actions)
├── Poller (4-Phasen, systemd Timer)
├── PostgreSQL
├── Caddy
└── Gateway (mit /ready + /health + X-Served-By)

DEV = Feature-Branches auf PROD-Forgejo
      (kein eigener Server!)

IAT = Temporärer Server, nur beim Release
      (gleiche Größe wie PROD)

FALLBACK = Temporärer Server, nur während Blue-Green Switch
           (PROD-Backup, für Rollback)
```

### UpCloud Resource Tagging

Jeder Server bekommt bei Erstellung Tags:

```yaml
# server-Rolle: create
body:
  server:
    title: "factory-{{ server_role }}-{{ run_id }}"
    tags:
      - "codefabrik"
      - "env:{{ server_role }}"         # prod / iat / fallback
      - "run:{{ run_id }}"              # 20260302-1430
      - "version:{{ release_version }}"  # v0.4.0
```

**Cleanup-Regeln:**
- **Standard-Delete** (Blue-Green, einzelner Run): löscht Server mit
  `codefabrik` AND `env:<role>` AND `run:<run_id>`. Nie nach IP, nie nach UUID.
- **Teardown (wipe_all):** löscht ALLE Server mit `codefabrik` Tag.
  Nur mit explizitem `--extra-vars wipe_all=true`. Schützt vor versehentlichem
  Löschen eines noch aktiven PROD bei mehreren parallelen Runs.

### Warum kein DEV-Server?

Die Fabrik entwickelt auf PROD. Der Entwicklungsprozess ist:

```
Story → PROD-Forgejo → PROD-Runner → Claude Code → PR → Tests → Review → Merge
```

DEV ist ein Feature-Branch, kein Server. Die Fabrik ist ihr eigenes Produkt
und behandelt sich wie jedes andere Produkt: Branches, PRs, Merges.

**Bekanntes Risiko (Reviewer):** Die Fabrik modifiziert sich selbst auf PROD.
Ein Bug in Poller/Bridge kann das laufende System beeinträchtigen. Mitigation:
- Kill Switch (fail-closed) begrenzt den Blast Radius
- Blue-Green: Infrastruktur-Änderungen werden auf IAT getestet
- v0.5.0 führt Control Plane Trennung ein (factory-core nur via Blue-Green)

### Version Pinning

Alle Komponenten mit festen Versionen:

```yaml
# inventory/group_vars/all/vars.yml
versions:
  forgejo: "10.0.3"
  forgejo_image: "codeberg.org/forgejo/forgejo:10.0.3@sha256:..."
  act_runner: "0.2.11"
  caddy_image: "caddy:2.9.1-alpine@sha256:..."
  postgres_image: "postgres:16.6-alpine@sha256:..."
  node_image: "node:22-slim@sha256:..."
  docker_compose: "2.32.4"
```

Alle Docker-Images mit Digest-Pin. Upgrade = bewusste Änderung in `vars.yml`
→ diff sichtbar → Blue-Green getestet. Kein Random-Update beim nächsten Build.

**Upgrade-Prozess (Runbook):**
1. Bump Version + Digest in `vars.yml` (z.B. Forgejo 10.0.3 → 10.0.4)
2. Commit + Push
3. `ansible-playbook blue-green.yml` → neues Release mit neuer Version auf IAT
4. Smoke-Test verifiziert neue Version (`X-Served-By` enthält neue Version)
5. DNS-Switch → neue Version ist PROD
6. Altes PROD als `pending_delete` (Rollback möglich für 60 Min)

### Kosten

DEV-1xCPU-4GB (40GB HDD):
- Running: ~15 EUR/Monat
- Stopped: ~4 EUR/Monat (nur Storage)
- Blue-Green Peak: 3x Running für ~10-15 Minuten (vernachlässigbar)

## /ready Endpoint (Gateway)

Der `/ready`-Endpoint ist die **harte Voraussetzung** für jeden DNS-Switch.
Ohne 200 von `/ready` wird kein Switch durchgeführt.

```
GET /ready
Response (200 NUR wenn alles ok UND mode=LIVE):
{
  "env": "prod",
  "version": "v0.4.0-abc1234",
  "served_by": "factory-prod",
  "mode": "LIVE",
  "traffic_enabled": true,
  "deps": {
    "forgejo": "ok",
    "postgres": "ok",
    "runner": "ok",
    "poller": "ok",
    "openclaw": "ok"
  }
}

Response Header: X-Served-By: prod:factory-prod:v0.4.0-abc1234

GET /health → 200 immer (Liveness, für Docker/systemd Health Checks)
```

**Blue-Green Regel:** DNS-Switch ist NUR erlaubt wenn:
1. `/ready` auf IAT gibt 200
2. `X-Served-By` enthält `iat` (nicht `prod`)

**Post-Switch Verifikation:**
1. `/ready` auf neuer DNS gibt 200
2. `X-Served-By` enthält `iat` (beweist: Traffic landet korrekt, kein DNS-Cache)

## Blue-Green Release — Detailablauf

### Phase 1: Vorbereitung

```
1. Kill Switch → mode=MAINTENANCE, jobs_enabled=false (traffic bleibt an)
2. Poller stoppt (prüft jobs_enabled)
3. Runner stoppen: systemctl stop act_runner (keine neuen CI-Writes)
4. Laufende Jobs abwarten (max 5 Min)
5. Forgejo graceful shutdown (keine Writes mehr)
6. pg_dump + rsync /data/gitea/repositories
7. Backup-Verifikation (dump size, repo count, git fsck Stichprobe)
8. Forgejo + Runner wieder starten (UI erreichbar bis Switch)
```

**Vollständiger Write-Freeze:** `jobs_enabled=false` stoppt den Poller, aber
nicht CI-Runner-Writes (Artifacts, Statuses) oder manuelle Git-Pushes.
Deshalb der 3-stufige Freeze:
1. Poller aus (jobs_enabled=false)
2. Runner aus (systemctl stop act_runner → keine laufenden Actions mehr)
3. Forgejo aus (`forgejo manager shutdown` → keine Git/API-Writes)

Erst dann ist pg_dump + rsync konsistent. Dauert wenige Sekunden,
Forgejo + Runner starten danach sofort wieder.

**Backup-Verifikation (4 Checks):**
- `pg_dump` Datei > 1 KB
- Repo-Verzeichnisse >= 2 (process-repo + mind. 1 Produkt)
- `git fsck --no-full` auf Stichprobe fehlerfrei
- Nach Restore: Forgejo API erreichbar + Auth ok

**Backup-Scope v0.4.0:** `pg_dump` + `rsync` der Repositories.
Kein vollständiger `gitea dump` (LFS, Attachments, Avatare → v0.5.0).
Das heißt: Rollback garantiert nur Repo + DB, nicht Dateianhänge.

### Phase 2: FALLBACK hochfahren

```
UpCloud API → neuer Server (gleiche Größe wie PROD, mit Tags)
ansible-playbook install.yml --limit fallback
Forgejo Dump einspielen + Restore-Verifikation
→ FALLBACK = Kopie von PROD (Rollback-Ziel)
```

Timeout: 5 Minuten für Server-Erstellung. Bei Timeout → Abbruch + PROD wieder an.

### Phase 3: IAT hochfahren

```
UpCloud API → neuer Server (mit Tags)
ansible-playbook install.yml --limit iat
Forgejo Dump einspielen + Restore-Verifikation
Neues Release deployen
```

### Phase 4: Smoke-Test auf IAT

```
Smoke-Test Checkliste:
  [] Forgejo API erreichbar + Auth ok
  [] Runner aktiv (act_runner → online)
  [] OpenClaw antwortet (docker exec factory-openclaw echo alive)
  [] PostgreSQL healthy (pg_isready)
  [] Poller läuft (systemctl is-active openclaw-poller.timer)
  [] Kill Switch funktioniert (jobs_enabled toggle)
  [] Caddy HTTPS funktioniert
  [] Gateway /ready gibt 200 (mode=LIVE, traffic_enabled=true)
  [] Gateway /health gibt korrekten Status
  [] X-Served-By enthält "iat"
  [] Dummy-Story durch Pipeline schieben (Inbox → Backlog → Branch → PR)
```

Alle Checks mit Retries/Backoff (Bootzeiten!), klare Timeouts.
Dummy-Story muss idempotent sein (cleanup nach Test).

Ergebnis:
- **Alle Checks grün** → `SMOKE_SUCCESS=true` → Switch erlaubt
- **Ein Check rot** → `SMOKE_SUCCESS=false` → Abbruch

### Phase 5a: Smoke-Test GRÜN → Switch

```
Cloudflare DNS → IAT-IP statt PROD-IP
Kill Switch auf neuem PROD → mode=LIVE, jobs_enabled=true, traffic_enabled=true
Warte auf DNS-Propagierung (60s)
Post-Switch Verifikation:
  → /ready auf neuer DNS gibt 200?
  → X-Served-By enthält "iat"? (beweist: Traffic landet korrekt)
Wenn Post-Switch grün → alte Server als "pending_delete" taggen
Wenn Post-Switch rot → Phase 5c (Rollback)
```

**Delay Delete:** Alte Server werden NICHT sofort gelöscht. Sie bekommen Tag
`pending_delete` + `delete_after_<timestamp>`. Manuelles Löschen nach 60 Min
oder per Cleanup-Job (v0.5.0). Safety-Netz gegen vorschnelle Löschung.

### Phase 5b: Smoke-Test ROT → Abbruch

```
IAT abreißen (per Tag)
FALLBACK abreißen (per Tag)
Kill Switch auf PROD → mode=LIVE, jobs_enabled=true
Notification: "Blue-Green abgebrochen, PROD unberührt"
PROD läuft weiter wie vorher
```

### Phase 5c: Verifizierung nach Switch FEHLGESCHLAGEN → Rollback

```
DNS → FALLBACK-IP
Smoke-Test auf FALLBACK (Go/No-Go)
→ FALLBACK grün:
    FALLBACK = neues PROD
    ex-IAT + altes PROD als pending_delete

→ FALLBACK auch rot:
    DNS → altes PROD IP (bekannt, noch nicht gelöscht dank Delay Delete)
    Kill Switch auf altes PROD → mode=LIVE
    Notification: "NOTFALL-ROLLBACK auf altes PROD"
    ex-IAT + FALLBACK als pending_delete
```

**Kurzzeitig 3 Server** (PROD + FALLBACK + IAT), danach wieder 1.
Maximale Sicherheit, minimale Kosten.

### Warum FALLBACK nötig?

Ohne FALLBACK: Wenn wir DNS von PROD auf IAT schwenken und IAT hat Probleme,
müssten wir auf PROD zurückschwenken. Aber PROD ist auch die Entwicklungsumgebung —
wenn wir es abreißen bevor IAT verifiziert ist, verlieren wir alles.

Mit FALLBACK: Egal was passiert, es gibt immer einen funktionierenden Server
zum Zurückschwenken. Der Zwischenzustand (3 Server) dauert nur Minuten.

## Nacht-Stopp

Solange kein Kunde auf der Fabrik arbeitet, wird nachts gestoppt:

```
22:00  Cronjob oder ansible-playbook nacht-stopp.yml --tags=stop
       1. Kill Switch → mode=MAINTENANCE, jobs_enabled=false
       2. Laufende Jobs abwarten (max 5 Min)
       3. UpCloud API → Server STOPPEN (nicht löschen!)
       4. Notification: "Server gestoppt"
       → Block Storage bleibt persistent
       → Docker-Volumes bleiben
       → Kosten: nur Storage, kein Compute

08:00  Cronjob oder ansible-playbook nacht-stopp.yml --tags=start
       1. UpCloud API → Server STARTEN
       2. Docker startet automatisch (restart: unless-stopped)
       3. act_runner startet automatisch (systemd enable)
       4. Poller startet automatisch (systemd Timer enable)
       5. Kill Switch → mode=LIVE, jobs_enabled=true, traffic_enabled=true
       6. Notification: "Server gestartet, /ready ok" oder "FEHLER: Start fehlgeschlagen"
       → Fabrik arbeitet weiter wo sie aufgehört hat
```

**Race Condition beim Start (gewollt, dokumentiert):**
Beim Start (08:00) passiert: Docker startet Container, Poller-Timer feuert,
aber `jobs_enabled` ist noch `false` (wurde beim Stop gesetzt). Der Poller
ist fail-closed und wird skippen bis Ansible `jobs_enabled=true` setzt.
Das ist **korrekt und sicher**. Die Reihenfolge ist:
1. Server starten → Docker + systemd starten automatisch
2. Poller feuert, sieht `jobs_enabled=false`, skippt (korrekt!)
3. Ansible setzt `mode=LIVE`, `jobs_enabled=true`
4. Nächster Poller-Run arbeitet normal

Kein `install.sh`/`teardown.sh` nötig — nur Server an/aus. Storage persistent.

**Übergang zu 24/7:** Wenn der erste Kunde drauf ist, Nacht-Stopp deaktivieren.
Alles andere bleibt gleich.

## Vier Stop-Szenarien

| Szenario | Kill Switch | Server Stop | DNS Switch | Server Delete |
|---|---|---|---|---|
| **Nacht-Stopp** | mode=OFF (Server geht aus) | Server stop | — | — |
| **Kill Switch** (Pause) | mode=MAINTENANCE (traffic bleibt) | — | — | — |
| **Wartungs-Stopp** | mode=OFF (Server geht aus) | Server stop | — | — |
| **Blue-Green Release** | mode=MAINTENANCE (alt, traffic bleibt) | — | DNS switch | pending_delete (alt+fallback) |

### Kill Switch (Sofort-Pause)

```bash
# Fabrik pausieren
ansible-playbook fabrik.yml --tags=kill-switch-off
# → jobs_enabled = false, Poller stoppt, laufende Jobs laufen aus

# Fabrik fortsetzen
ansible-playbook fabrik.yml --tags=kill-switch-on
# → jobs_enabled = true
```

### Wartungs-Stopp

```bash
ansible-playbook nacht-stopp.yml --tags=stop
# Wartung durchführen (z.B. via SSH)
ansible-playbook nacht-stopp.yml --tags=start
```

## Kill-Switch-Schema

Erweitertes Schema, adaptiert von studio-ops. Ersetzt das einfache `key/value`
aus v0.3.1.

**Warum die Erweiterung nötig ist:**
- Blue-Green Phase 1: `jobs_enabled=false` auf PROD, aber Forgejo UI soll
  noch erreichbar sein → `traffic_enabled` separat
- Mehrere Server: PROD, IAT, FALLBACK brauchen eigene Einträge → `env_name`
- Debugging: `reason` + `updated_by` + `updated_at` zeigen wer wann warum umgeschaltet hat
- `/ready`-Gate: Braucht `mode` für programmatische Entscheidung (200 nur bei LIVE)
- Optimistic Locking: `generation` verhindert Race Conditions bei concurrent Updates

```sql
CREATE TABLE IF NOT EXISTS runtime_control (
    env_name        TEXT PRIMARY KEY,  -- 'prod', 'iat', 'fallback'
    mode            TEXT NOT NULL DEFAULT 'PRELIVE'
                    CONSTRAINT valid_mode CHECK (mode IN ('OFF', 'PRELIVE', 'LIVE', 'MAINTENANCE')),
    traffic_enabled BOOLEAN NOT NULL DEFAULT FALSE,
    jobs_enabled    BOOLEAN NOT NULL DEFAULT FALSE,
    reason          TEXT NOT NULL DEFAULT '',
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_by      TEXT NOT NULL DEFAULT '',
    generation      BIGINT NOT NULL DEFAULT 0
);

-- Fail-Closed Default: PRELIVE mit beiden Flags auf false
INSERT INTO runtime_control (env_name, mode, reason, updated_by)
VALUES ('prod', 'PRELIVE', 'Initial Setup', 'install.yml')
ON CONFLICT DO NOTHING;
```

### Mode-Semantik (Policy)

| Mode | traffic_enabled | jobs_enabled | Wann |
|---|---|---|---|
| `OFF` | false | false | Server existiert, aber komplett inaktiv |
| `PRELIVE` | false | false | Frisch installiert, noch nicht validiert |
| `LIVE` | true | true | Normalbetrieb |
| `MAINTENANCE` | **true** | false | Wartung / Blue-Green Freeze |

**Wichtig: MAINTENANCE ≠ alles zu.** In MAINTENANCE bleibt `traffic_enabled=true`:
Forgejo UI, Git-Klonen, SSH-Zugriff, PRs anschauen — alles erreichbar.
Nur die Fabrik-Pipeline stoppt (`jobs_enabled=false`).

**Was steuert was:**
- `traffic_enabled` → Gateway/Caddy Routing für **Factory-Endpoints** (API, Webhooks)
- Forgejo-Web/Git-Access → immer an, solange Server läuft (unabhängig von `traffic_enabled`)
- `jobs_enabled` → Poller, OpenClaw, neue Pipeline-Runs

**Regel:** `/ready` gibt 200 NUR bei `mode = LIVE`.
In MAINTENANCE gibt `/ready` 503 (Factory nicht bereit), aber `/health` bleibt 200 (Server lebt).
Ansible-Playbooks setzen `mode` explizit (nicht nur die Flags).

### Optimistic Locking

Alle Kill-Switch-Updates laufen über die `fabrik`-Rolle (`set-mode` Task),
nie direkt per Shell/Heredoc. Die Rolle:

1. `SELECT generation FROM runtime_control WHERE env_name='$ENV'`
2. `UPDATE ... SET mode=..., generation=generation+1 WHERE env_name='$ENV' AND generation=$EXPECTED`
3. Prüft `rows affected == 1` (sonst → concurrent modification → fail)

```sql
-- Beispiel (wird von fabrik-Rolle generiert, nie manuell):
UPDATE runtime_control
SET mode = 'LIVE',
    jobs_enabled = true,
    traffic_enabled = true,
    reason = 'Blue-Green Switch done',
    updated_by = 'blue-green.yml',
    updated_at = now(),
    generation = generation + 1
WHERE env_name = 'prod'
  AND generation = 42;
-- 0 rows affected = concurrent modification → Fehler!
```

Implementierung in der Rolle: `docker exec factory-postgres psql -U forgejo -d forgejo
-v ON_ERROR_STOP=1 -tAq -c "<single statement>"` (separate Aufrufe, kein Heredoc).

### Fail-Closed Prinzip

1. Default: PRELIVE, traffic=false, jobs=false
2. DB nicht erreichbar → fail-closed (kein Run, kein Traffic)
3. Fehlende Row → fail-closed
4. Poller prüft `jobs_enabled`
5. Gateway prüft `traffic_enabled` + `mode` via `/ready`

**Poller-Anpassung:**

```bash
JOBS_ENABLED=$(db_query "SELECT jobs_enabled FROM runtime_control WHERE env_name='$ENV_NAME'")
# PostgreSQL gibt Booleans als 't'/'f' zurück
if [ "$JOBS_ENABLED" != "t" ]; then
  log "SKIP: jobs disabled (jobs_enabled=$JOBS_ENABLED)"
  exit 0
fi
```

## Testkonzept (minimal für v0.4.0)

### Grundprinzip: Alles ist ein Produkt

Es gibt keine Unterscheidung zwischen "Infrastruktur" und "Produkt".
Alles ist ein Produkt. Das Testverfahren skaliert mit dem **Blast Radius**:

| Blast Radius | Komponenten | Testverfahren |
|---|---|---|
| **Hoch** | Forgejo, OpenClaw, Poller, PostgreSQL, Runner | Blue-Green auf IAT |
| **Niedrig** | Gateway, zukünftige Produkte | Feature-Branch + Tests auf PROD |

### Smoke-Test (Go/No-Go Gate)

Der Smoke-Test läuft automatisch nach jedem Deployment und entscheidet
ob ein Blue-Green Switch durchgeführt wird:

```
Smoke-Test Checkliste:
  [] Forgejo API erreichbar + Auth ok
  [] Runner aktiv (act_runner list → online)
  [] OpenClaw antwortet (docker exec factory-openclaw echo alive)
  [] PostgreSQL healthy (pg_isready)
  [] Poller läuft (systemctl is-active openclaw-poller.timer)
  [] Kill Switch funktioniert (jobs_enabled toggle)
  [] Caddy HTTPS funktioniert
  [] Gateway /ready gibt 200 (mode=LIVE, traffic_enabled=true)
  [] Gateway /health gibt korrekten Status
  [] X-Served-By korrekt
  [] Dummy-Story durch Pipeline schieben (Inbox → Backlog → Branch → PR)
```

Alle Checks mit Retries/Backoff (Bootzeiten!) und klaren Timeouts.
`/ready` ist das **programmatische Gate** — nicht nur eine Checkliste.

**Dummy-Story Konvention:**
- Story-ID fest: `SMOKE-000` (nie produktive IDs)
- Branch-Prefix: `smoke/SMOKE-000`
- PR-Label: `smoke-test`
- Cleanup nach Test: PR schließen + Branch löschen + Story zurückräumen
- Während Blue-Green Freeze ist Poller auf PROD aus → der Smoke-Test
  auf IAT muss dort den Poller gezielt aktivieren (Kill Switch lokal auf
  IAT → LIVE) oder die Story direkt per API/Script verarbeiten.

Ergebnis:
- **Alle Checks grün** → `SMOKE_SUCCESS=true` → Blue-Green Switch erlaubt
- **Ein Check rot** → `SMOKE_SUCCESS=false` → Abbruch, kein Switch

### Späterer Ausbau (nicht v0.4.0)

- Integrationstests (mehrstufige Stories, komplexe Workflows)
- Performance-Tests (Durchsatz messen, Laufzeiten vergleichen)
- Rollback-Tests (absichtlich kaputt machen, Rollback verifizieren)
- Automatische Blast-Radius-Klassifizierung pro Produkt

## Minimal Notification

Ein einziger Webhook-Kanal für alle Benachrichtigungen.
Kein Monitoring-Stack, kein Dashboard — nur "ich merke es zuverlässig".

```yaml
# roles/notify/tasks/main.yml
- name: Notification senden
  ansible.builtin.uri:
    url: "{{ vault_notify_webhook_url }}"
    method: POST
    body_format: json
    body:
      title: "{{ notify_title }}"
      message: "{{ notify_message }}"
      priority: "{{ notify_priority | default('default') }}"
      tags: ["codefabrik", "{{ notify_action }}"]
  ignore_errors: true  # Notification-Fehler darf Run nicht abbrechen
```

**Webhook-Ziel:** ntfy.sh (self-hosted oder public), Pushover, oder SMTP.
Konfiguriert in `vault.yml` als `vault_notify_webhook_url`.

**Eingebaut in:**
- `blue-green.yml`: rescue (Fehler) + always (Ergebnis)
- `nacht-stopp.yml`: nach Start-Erfolg, nach Start-Fehler, nach Stop
- `install.yml`: nach Abschluss
- `teardown.yml`: nach Abschluss

**Externes Uptime-Monitoring:** Zusätzlich ein externer Dienst (z.B. UptimeRobot),
der alle 1-5 Minuten `/ready` prüft. Fängt ab: Nacht-Start fehlgeschlagen,
Caddy down, DNS zeigt ins Nichts.

## Dokumenten-Lifecycle (Minimal für v0.4.0)

### Grundprinzip

Die Code-Fabrik bekommt nicht nur eine Story-Idee, sondern **alle Release-relevanten
Dokumente** als Übergabepaket. Nach der Verarbeitung werden die erzeugten Artefakte
strukturiert abgelegt — **immer zuordenbar zu Anforderung und Release**.

### v0.4.0: Minimale Umsetzung

Drei neue YAML-Felder in der Story + eine Index-Datei pro Release:

```yaml
# Story-YAML (process-repo: work/01-inbox/STORY-000123.yml)
produkt: factory-gateway
release: v0.4.0          # NEU: Ziel-Release
scope_in:                  # NEU: Was ist drin
  - "/ready Endpoint"
  - "X-Served-By Header"
scope_out:                 # NEU: Was explizit nicht
  - "Dashboard UI"
```

```
# Ablage pro Release (im Produkt-Repository)
factory-gateway/
└── docs/
    └── releases/
        └── v0.4.0/
            └── stories.md    ← Index: welche Stories in diesem Release
```

Poller-Bridge legt `_task.md` ab (existiert schon, keine Änderung nötig).

### v0.5.0: Volle Dokumentenablage

Erst wenn die Pipeline stabil läuft:

## Ansible-Architektur

### Warum Ansible?

v0.3.0/v0.3.1 nutzt einen riesigen Claude-Code-Prompt der via SSH alles sequentiell
abarbeitet. Das hat Nachteile:

- Nicht wiederholbar (Prompt wird bei jedem Lauf neu interpretiert)
- Nicht idempotent (zweiter Lauf kann Dinge doppelt anlegen)
- Keine Trennung von Config und Logik
- Secrets in `.env`-Dateien statt in einem Vault
- Kein Diff möglich (was hat sich geändert?)

Ansible löst das:

- **Deklarativ:** Zustand beschreiben statt Schritte ausführen
- **Idempotent:** Kann beliebig oft laufen, gleiches Ergebnis
- **Vault:** Secrets verschlüsselt im Repo
- **Rollen:** Wiederverwendbare Bausteine
- **Inventory:** PROD/IAT/FALLBACK als Targets

### Ansible Vault — Secrets

```bash
# Vault erstellen
ansible-vault create inventory/group_vars/all/vault.yml

# Inhalt (verschlüsselt):
vault_anthropic_api_key: sk-ant-...
vault_upcloud_api_token: ucat_...
vault_cloudflare_api_token: ...
vault_ollama_api_key: ...
vault_notify_webhook_url: https://ntfy.sh/...

# Ausführen
ansible-playbook install.yml --ask-vault-pass
# Oder mit Vault-Password-File:
ansible-playbook install.yml --vault-password-file ~/.vault_pass
```

Keine `.env-codefabrik` mehr. Alle Secrets im Vault, verschlüsselt im Repo.

**SSH Deploy-Key Härtung:**
- `~/.ssh/codefabrik_deploy` (Ed25519, ohne Passphrase) — nur für Automation
- `authorized_keys` auf Server: `no-agent-forwarding,no-port-forwarding,no-pty`
  (eingeschränkt, da Ansible nur `command` + `shell` braucht)
- `~/.vault_pass` Permissions: `chmod 0600`, Pfad in `ansible.cfg` dokumentiert
- Nicht dein persönlicher SSH-Key — separater Key nur für Code-Fabrik

**Secrets-Rotation bei Blue-Green:** Forgejo API-Tokens werden bei jedem Deploy
neu generiert (neuer Server = neuer Admin-Account = neuer Token).
Die `forgejo`-Rolle: (1) Admin-Account anlegen, (2) neuen Token generieren,
(3) Token auf Server + in Forgejo Secrets aktualisieren.
Der Vault enthält nur **externe** Secrets (Anthropic, UpCloud, Cloudflare, Ollama).
Forgejo-Tokens sind deploy-time generiert, nicht im Vault.

### blue-green.yml — Release-Prozess

```yaml
# Blue-Green Release mit 3 Servern
# Aufruf: ansible-playbook blue-green.yml
- name: "Phase 0: Preflight"
  hosts: localhost
  tasks:
    - name: UpCloud/DNS Credentials prüfen
      ansible.builtin.uri:
        url: "https://api.upcloud.com/1.3/account"
        headers:
          Authorization: "Bearer {{ vault_upcloud_api_token }}"
      register: preflight_upcloud
      failed_when: preflight_upcloud.status != 200

- name: "Phase 1: Vorbereitung"
  hosts: prod
  tasks:
    - name: Kill Switch → MAINTENANCE (traffic bleibt an, jobs aus)
      include_role: { name: fabrik, tasks_from: kill-switch-maintenance }
    - name: Runner stoppen (keine CI-Writes mehr)
      ansible.builtin.shell: systemctl stop act_runner
    - name: Laufende Jobs abwarten
      include_role: { name: fabrik, tasks_from: wait-for-idle }
    - name: Forgejo herunterfahren (Write-Freeze)
      ansible.builtin.shell: >
        docker exec -u git factory-forgejo forgejo manager shutdown
    - name: Forgejo Dump erstellen
      include_role: { name: backup, tasks_from: dump }
    - name: Backup verifizieren
      include_role: { name: backup, tasks_from: verify }
    - name: Forgejo + Runner wieder starten
      ansible.builtin.shell: |
        docker start factory-forgejo
        systemctl start act_runner

- name: "Phase 2: FALLBACK hochfahren"
  hosts: localhost
  tasks:
    - name: FALLBACK Server erstellen
      block:
        - include_role: { name: server, tasks_from: create }
          vars: { server_role: fallback, server_timeout: 300 }
      rescue:
        - include_role: { name: fabrik, tasks_from: kill-switch-on }
          delegate_to: factory-prod
        - include_role: { name: notify }
          vars:
            notify_title: "Blue-Green ABBRUCH"
            notify_message: "FALLBACK-Server konnte nicht erstellt werden"
            notify_action: bluegreen
            notify_priority: high
        - fail:
            msg: "Server-Erstellung fehlgeschlagen"

- name: "Phase 2b: FALLBACK installieren"
  hosts: fallback
  roles: [base, postgres, caddy, forgejo, runner, openclaw, gateway, poller, repos, secrets]
  post_tasks:
    - name: Backup einspielen
      include_role: { name: backup, tasks_from: restore }
    - name: Restore verifizieren
      include_role: { name: backup, tasks_from: verify-restore }

- name: "Phase 3: IAT hochfahren"
  hosts: localhost
  tasks:
    - name: IAT Server erstellen
      block:
        - include_role: { name: server, tasks_from: create }
          vars: { server_role: iat, server_timeout: 300 }
      rescue:
        - include_role: { name: server, tasks_from: delete }
          vars: { server_role: fallback }
        - include_role: { name: fabrik, tasks_from: kill-switch-on }
          delegate_to: factory-prod
        - include_role: { name: notify }
          vars:
            notify_title: "Blue-Green ABBRUCH"
            notify_message: "IAT-Server konnte nicht erstellt werden"
            notify_action: bluegreen
            notify_priority: high
        - fail:
            msg: "Server-Erstellung fehlgeschlagen"

- name: "Phase 3b: IAT installieren + Release deployen"
  hosts: iat
  roles: [base, postgres, caddy, forgejo, runner, openclaw, gateway, poller, repos, secrets]
  post_tasks:
    - name: Backup einspielen
      include_role: { name: backup, tasks_from: restore }
    - name: Restore verifizieren
      include_role: { name: backup, tasks_from: verify-restore }
    - name: Neues Release deployen
      include_role: { name: release, tasks_from: deploy }

- name: "Phase 4: Smoke-Test auf IAT"
  hosts: iat
  roles:
    - smoke-test

- name: "Phase 5: Switch oder Rollback"
  hosts: localhost
  vars:
    # smoke_test_result wurde auf iat gesetzt → über hostvars holen
    smoke_success: "{{ hostvars['factory-iat'].smoke_test_result | default('failed') == 'passed' }}"
  tasks:
    - name: "5a: DNS Switch (wenn grün)"
      when: smoke_success
      block:
        - include_role: { name: dns, tasks_from: switch }
          vars: { new_target_ip: "{{ iat_server_ip }}" }
        - include_role: { name: fabrik, tasks_from: kill-switch-on }
          delegate_to: factory-iat

        - name: Warte auf DNS-Propagierung
          ansible.builtin.pause: { seconds: 60 }

        - name: Post-Switch Verifikation (via DNS — beweist dass Traffic korrekt landet)
          ansible.builtin.uri:
            url: "https://{{ domain }}/ready"
            return_content: true
          register: post_switch_dns
          retries: 5
          delay: 15
          until: >
            post_switch_dns.status == 200 and
            'iat' in (post_switch_dns.json.served_by | default(''))
          failed_when: >
            post_switch_dns.status != 200 or
            'iat' not in (post_switch_dns.json.served_by | default(''))

        - name: Post-Switch Verifikation (IP-direkt — unabhängig von DNS-Cache)
          ansible.builtin.uri:
            url: "https://{{ iat_server_ip }}/ready"
            validate_certs: false
            headers:
              Host: "{{ domain }}"
          register: post_switch_ip
          failed_when: post_switch_ip.status != 200

        - name: "5a: Alte Server als pending_delete taggen"
          when: post_switch_dns is success and post_switch_ip is success
          include_role: { name: server, tasks_from: tag-pending-delete }
          loop: [prod, fallback]

        - name: "5c: Rollback wenn Post-Switch fehlschlägt"
          when: post_switch_dns is failed or post_switch_ip is failed
          block:
            - name: Versuch 1 — Rollback auf FALLBACK
              block:
                - include_role: { name: dns, tasks_from: switch }
                  vars: { new_target_ip: "{{ fallback_server_ip }}" }
                - name: Smoke-Test auf FALLBACK
                  ansible.builtin.uri:
                    url: "https://{{ fallback_server_ip }}/ready"
                    validate_certs: false
                    headers: { Host: "{{ domain }}" }
                  retries: 3
                  delay: 10
                - include_role: { name: fabrik, tasks_from: kill-switch-on }
                  delegate_to: factory-fallback
                - include_role: { name: notify }
                  vars:
                    notify_title: "ROLLBACK auf FALLBACK"
                    notify_message: "Post-Switch fehlgeschlagen, FALLBACK übernimmt"
                    notify_action: bluegreen
                    notify_priority: high
              rescue:
                - name: Versuch 2 — Notfall-Rollback auf altes PROD
                  block:
                    - include_role: { name: dns, tasks_from: switch }
                      vars: { new_target_ip: "{{ old_prod_server_ip }}" }
                    - include_role: { name: fabrik, tasks_from: kill-switch-on }
                      delegate_to: factory-prod
                    - include_role: { name: notify }
                      vars:
                        notify_title: "NOTFALL-ROLLBACK auf altes PROD"
                        notify_message: "FALLBACK auch kaputt! Zurück auf altes PROD."
                        notify_action: bluegreen
                        notify_priority: urgent
            - fail:
                msg: "Blue-Green Switch fehlgeschlagen, Rollback durchgeführt"

    - name: "5b: Abbruch (wenn rot)"
      when: not smoke_success
      block:
        - include_role: { name: server, tasks_from: delete }
          vars: { server_role: iat }
        - include_role: { name: server, tasks_from: delete }
          vars: { server_role: fallback }
        - include_role: { name: fabrik, tasks_from: kill-switch-on }
          delegate_to: factory-prod

  post_tasks:
    - name: Ergebnis-Notification (immer)
      include_role: { name: notify }
      vars:
        notify_title: "Blue-Green {{ 'OK' if smoke_success else 'ABBRUCH' }}"
        notify_message: "Smoke: {{ smoke_test_result }}"
        notify_action: bluegreen
```

### nacht-stopp.yml — Server Start/Stop

```yaml
# Aufruf: ansible-playbook nacht-stopp.yml --tags=stop
#         ansible-playbook nacht-stopp.yml --tags=start
- name: Nacht-Stopp
  hosts: localhost
  tasks:
    - name: Server stoppen
      tags: [stop]
      block:
        - name: Kill Switch → OFF (Server geht gleich aus)
          include_role: { name: fabrik, tasks_from: set-mode }
          vars:
            target_host: "{{ prod_server_ip }}"
            target_mode: "OFF"
            target_jobs: false
            target_traffic: false
            target_reason: "Nacht-Stopp"
          # fabrik-Rolle führt aus:
          # 1. SELECT generation FROM runtime_control WHERE env_name='prod'
          # 2. UPDATE ... SET mode='OFF', jobs_enabled=false, traffic_enabled=false,
          #    generation=generation+1 WHERE env_name='prod' AND generation=$EXPECTED
          # 3. Prüft rows affected == 1 (sonst concurrent modification → fail)
        - name: Warte auf laufende Jobs
          ansible.builtin.pause:
            seconds: 300
            prompt: "Warte max 5 Min auf laufende Jobs..."
        - name: Server stoppen (UpCloud API)
          ansible.builtin.uri:
            url: "https://api.upcloud.com/1.3/server/{{ prod_server_uuid }}/stop"
            method: POST
            headers:
              Authorization: "Bearer {{ vault_upcloud_api_token }}"
              Content-Type: "application/json"
            body_format: json
            body:
              stop_server:
                stop_type: soft
                timeout: 120
        - name: Notification
          include_role: { name: notify }
          vars:
            notify_title: "Nacht-Stopp OK"
            notify_message: "Server gestoppt"
            notify_action: nacht-stopp
      rescue:
        - include_role: { name: notify }
          vars:
            notify_title: "Nacht-Stopp FEHLER"
            notify_message: "Stop fehlgeschlagen"
            notify_action: nacht-stopp
            notify_priority: high

    - name: Server starten
      tags: [start]
      block:
        - name: Server starten (UpCloud API)
          ansible.builtin.uri:
            url: "https://api.upcloud.com/1.3/server/{{ prod_server_uuid }}/start"
            method: POST
            headers:
              Authorization: "Bearer {{ vault_upcloud_api_token }}"
              Content-Type: "application/json"
            body_format: json
            body:
              start_server:
                avoid_host: 0
        - name: Warte auf SSH
          ansible.builtin.wait_for:
            host: "{{ prod_server_ip }}"
            port: 22
            timeout: 300
        - name: Kill Switch → LIVE
          include_role: { name: fabrik, tasks_from: set-mode }
          vars:
            target_host: "{{ prod_server_ip }}"
            target_mode: "LIVE"
            target_jobs: true
            target_traffic: true
            target_reason: "Nacht-Start"
        - name: /ready prüfen (IP-direkt, unabhängig von DNS)
          ansible.builtin.uri:
            url: "https://{{ prod_server_ip }}/ready"
            validate_certs: false
            headers:
              Host: "{{ domain }}"
          retries: 10
          delay: 15
        - name: Notification
          include_role: { name: notify }
          vars:
            notify_title: "Nacht-Start OK"
            notify_message: "Server gestartet, /ready grün"
            notify_action: nacht-start
      rescue:
        - include_role: { name: notify }
          vars:
            notify_title: "Nacht-Start FEHLER"
            notify_message: "Start fehlgeschlagen!"
            notify_action: nacht-start
            notify_priority: high
```

### teardown.yml — Kompletter Abriss

```yaml
# Aufruf: ansible-playbook teardown.yml [--extra-vars "backup=true"]
- name: Code-Fabrik abreißen
  hosts: localhost
  tasks:
    - name: Backup erstellen
      when: backup | default(false) | bool
      include_role: { name: backup, tasks_from: dump }

    - name: Server löschen (nur codefabrik-getaggte, mit wipe_all Guard)
      include_role: { name: server, tasks_from: delete-by-tag }
      vars:
        delete_filter_tags: "{{ ['codefabrik'] + ([] if (wipe_all | default(false) | bool) else ['run:' + run_id]) }}"
      ignore_errors: true
      # Standard: löscht nur Server des aktuellen Runs
      # Mit --extra-vars wipe_all=true: löscht ALLE codefabrik-Server

    - name: DNS aufräumen
      include_role: { name: dns, tasks_from: cleanup }

    - name: Lokale Dateien aufräumen
      ansible.builtin.file:
        path: "{{ item }}"
        state: absent
      loop:
        - ~/code-fabrik/.server-env
        - ~/code-fabrik/.tokens-env

  post_tasks:
    - name: Notification
      include_role: { name: notify }
      vars:
        notify_title: "Teardown abgeschlossen"
        notify_message: "Alle Server + DNS gelöscht"
        notify_action: teardown
```

## Blue-Green Failure Matrix (Runbook)

| Failure | Detection | Auto-Reaction | Manuelles Runbook |
|---|---|---|---|
| **UpCloud API down / Timeout** | `uri` mit retries, am Ende fail | Abbruch bevor PROD verändert | API Status prüfen, später erneut |
| **Server erstellt, SSH nicht erreichbar** | `wait_for: port=22` Timeout | Server löschen, Abbruch | UpCloud Console, Firewall prüfen |
| **Backup inkonsistent** | dump size, repo count, git fsck | Abbruch + PROD wieder an | Forgejo Shutdown härter, Backup erneut |
| **Restore fehlgeschlagen** | pg_restore return code, Forgejo API | Abbruch + Cleanup (IAT/FALLBACK) | Versions-Pinning prüfen, Migrations |
| **Runner registriert nicht** | Forgejo API Runner-Status | Smoke-Test fail → Abbruch | Token, clock skew, service logs |
| **Secrets fehlen** | Smoke-Test Action mit Secret-Referenz | Abbruch | Reihenfolge: secrets nach repos |
| **DNS Switch ok, aber Traffic auf alt** | `X-Served-By` Header prüfen | Warten + Retry, kein sofort-delete | TTL, Cloudflare Proxy Settings |
| **Post-Switch IAT kaputt** | Post-Switch Smoke fail | Rollback DNS → FALLBACK | FALLBACK Smoke prüfen |
| **FALLBACK auch kaputt** | Smoke auf FALLBACK fail | Auto-Rollback: DNS → altes PROD IP, Kill Switch LIVE | Prüfen warum beide kaputt, Dump/Restore debuggen |
| **Zombie-Server (partial success)** | `always`-Block: list servers by tag+run_id | Cleanup-Versuch + Notification mit Server-IDs | Manuell löschen via UpCloud Console |
| **Falscher Server gelöscht** | Delete-Guard: nur codefabrik+env+run_id Tags | Standard-Delete nie ohne run_id; wipe_all nur explizit | UpCloud Console, Server ggf. noch in stopped state |
| **Dummy-Story verschmutzt Repos** | SMOKE-000 Konvention, smoke/ Branch-Prefix | Cleanup-Task: PR schließen, Branch löschen, Story zurück | Manuell aufräumen, keine produktiven IDs betroffen |

## Lessons Learned aus v0.3.0/v0.3.1

Diese Erkenntnisse sind in die v0.4.0 Architektur eingeflossen:

| Erkenntnis | Auswirkung auf v0.4.0 |
|---|---|
| Forgejo hat kein `repository_dispatch` | Poller-basierte Pipeline bleibt |
| Forgejo unterdrückt rekursive Action-Triggers | Actions nur für native Events (PR, Push) |
| `act_runner` als Host-Service stabiler als Container | Runner-Rolle installiert Host-Service |
| `psql -tAq` statt `psql -t` (Multi-Statement-Bug) | In Poller-Template korrigiert |
| `docker exec -u git` für Forgejo Admin-Befehle | In Forgejo-Rolle berücksichtigt |
| OpenClaw Base-Image 403 | Eigenes Image mit `node:22-slim` (version-pinned) |
| SSH-Key ohne Passphrase nötig für Automation | `codefabrik_deploy` Key (Ed25519) |
| Claude Code Memory kann Install verwirren | Ansible statt Claude-Code-Prompt |
| UpCloud Plan-Namen ändern sich | Dynamische Plan-Abfrage in Server-Rolle |
| `max_turns` Limit bei komplexen Installationen | Ansible hat kein Turn-Limit |
| DB-Tabellen: Heredoc über SSH+docker verschluckt Statements | `postgres`-Rolle: separate SQL-Dateien |
| Lock-File `kill -0` crasht bei `set -euo pipefail` | Poller-Template: defensive Absicherung |
| `.server-env`/`.tokens-env` überleben Teardown | Teardown räumt lokale State-Dateien auf |
| `act_runner register`: `token is empty` bei env-Token | Runner-Rolle: Token als CLI-Flag `--token` |
| Install-Prompt wird als Frage interpretiert | Ansible hat dieses Problem nicht (deklarativ) |

## ADRs (neu in v0.4.0)

### ADR-007: Poller statt Workflow-Ketten

Forgejo 10.x unterdrückt rekursive Workflow-Triggers und hat kein
`repository_dispatch`. Der Poller übernimmt dispatch + bridge +
openclaw-trigger + pr-creation. Forgejo Actions nur für native
Events: `pull_request` (auto-test) und `push` auf main (auto-release).

### ADR-008: Ansible statt Claude-Code-Prompt

v0.3.0 nutzt einen 2400-Zeilen Claude-Code-Prompt für die Installation.
Probleme: nicht wiederholbar, nicht idempotent, max_turns Limit,
Memory-Interferenz. v0.4.0 migriert auf Ansible Playbooks + Vault.

### ADR-009: Blue-Green mit drei Servern

Beim Release-Switch wird kurzzeitig ein FALLBACK-Server hochgefahren
um die Entwicklungsumgebung (= PROD) nicht zu verlieren. Ohne FALLBACK:
kein Rollback-Ziel wenn das neue PROD Probleme hat.

### ADR-010: Alles ist ein Produkt

Keine Unterscheidung Infrastruktur vs. Produkt. Testverfahren skaliert
mit Blast Radius: hoher Blast Radius (Forgejo, OpenClaw, Poller) →
Blue-Green auf IAT. Niedriger Blast Radius (Gateway etc.) →
Feature-Branch + Tests auf PROD.

### ADR-011: Nacht-Stopp statt Teardown

Solange nicht produktiv: Server nachts stoppen (UpCloud API stop/start),
nicht abreißen. Block Storage + Docker Volumes bleiben persistent.
Spart Compute-Kosten, vermeidet täglichen Neuaufbau.

### ADR-012: Kill-Switch-Schema

v0.3.1 nutzt ein simples key/value-Schema. v0.4.0 übernimmt das bewährte
studio-ops Schema mit `env_name`, `mode`, `traffic_enabled`, `jobs_enabled`,
`generation` (optimistic locking), `reason` + `updated_by` (Audit-Trail).
Fail-Closed Default. `/ready` gibt 200 nur bei `mode=LIVE`.

### ADR-013: Delay Delete

Beim Blue-Green Switch werden alte Server nicht sofort gelöscht, sondern
als `pending_delete` getaggt. Manuelles Löschen nach 60 Minuten.
Begründung: Safety-Netz gegen vorschnelle Löschung, DNS-Propagierung,
unerwartete Probleme nach Switch.

## Umsetzung: Sprints

```
Sprint 1: Ansible Foundation (2 Wochen)
  Rollen: server (mit Tagging), base, dns, postgres, forgejo,
          runner, openclaw, caddy, gateway, poller, repos, secrets
  Playbooks: install.yml, teardown.yml
  Härtungen: UpCloud Tagging, Version Pinning
  Validierung: teardown + install + /ready 200

Sprint 2: Betrieb (1 Woche)
  Rollen: smoke-test (mit /ready + X-Served-By), notify
  Playbooks: nacht-stopp.yml, fabrik.yml (Kill Switch)
  Härtungen: /ready Endpoint, Minimal Notification
  Erweiterung: Kill Switch mit generation, Mode-Semantik
  Validierung: Stop → Start → /ready 200 → Notification erhalten

Sprint 3: Blue-Green (2 Wochen)
  Rollen: backup (mit Verifikation), release
  Playbook: blue-green.yml (mit block/rescue/always)
  Härtungen: Backup-Verifikation, Delay Delete, Forgejo Shutdown bei Dump
  Erweiterung: Failure Matrix als Runbook-Dokument
  Validierung: Vollständiger Blue-Green-Zyklus + Rollback-Test

Sprint 4: Docs + Stabilisierung (1 Woche)
  Dokumenten-Lifecycle (minimal): Story-YAML Felder, stories.md Index
  Blue-Green Failure Matrix als Runbook-Dokument finalisieren
  Externes Uptime-Monitoring einrichten (z.B. UptimeRobot auf /ready)
  End-to-End: install → blue-green → nacht-stopp → teardown
```

## v0.5.0 — "Wachposten"

> v0.4.0 = Schichtbetrieb (Fabrik läuft zuverlässig, Blue-Green, Nacht-Stopp)
> v0.5.0 = Wachposten (Fabrik überwacht sich selbst, separiert Control Plane, sichert Backups ab)

### Leitprinzip: "Never trust the worker"

Die Code-Fabrik ist ein self-modifying system. Der Worker (OpenClaw/Claude Code) hat
Schreibzugriff auf Repos und kann theoretisch auch Infrastruktur-Code ändern. v0.5.0
etabliert die Trennung zwischen **Control Plane** (Infrastruktur, Poller, Gateway) und
**Data Plane** (Produkt-Repos, Feature-Branches). Die Kernfrage jeder v0.5.0-Entscheidung:
*"Was passiert, wenn der Worker etwas Unerwartetes tut?"*

### Paket A — Briefkasten + Watchdog (MUSS)

**Ziel:** 24/7-erreichbarer Mini-Server als API-Eingang und Überwachungsinstanz.

| Komponente | Beschreibung |
|---|---|
| **Briefkasten-Server** | Persistenter 1GB-Server (UpCloud, 24/7). Läuft unabhängig vom Fabrik-Motor. |
| **POST /stories** | Nimmt Story-YAMLs entgegen, validiert Schema, legt in Queue ab. |
| **GET /queue** | Zeigt aktuelle Queue (pending, in-progress, done). |
| **POST /actions/start-worker** | Startet den Fabrik-Motor (UpCloud Server hochfahren + Ansible install.yml). |
| **POST /actions/kill-switch** | Setzt Kill Switch auf OFF für alle Envs. Notbremse via API. |
| **Watchdog** | Pollt /ready auf PROD (und ggf. IAT). Bei Ausfall → Notification + optionaler Auto-Kill-Switch. |
| **Webhook-Relay** | Nimmt eingehende Webhooks (z.B. Digistore24) an, queued sie, leitet weiter wenn Motor läuft. |

**Akzeptanztest Paket A:**
1. `curl -X POST /stories` mit gültiger YAML → 202 Accepted, Story in Queue
2. `curl -X POST /stories` mit ungültiger YAML → 400 mit Fehlerbeschreibung
3. Motor ist aus → `POST /actions/start-worker` → Server hochgefahren, Pipeline startet
4. Motor ist an, /ready antwortet nicht → Watchdog erkennt Ausfall innerhalb 5 Min, sendet Notification
5. `POST /actions/kill-switch` → Kill Switch auf OFF, Poller stoppt innerhalb 30s

### Paket B — Control Plane Trennung (MUSS)

**Ziel:** Self-Modification-Risiko entschärfen. Factory-Core darf nur via Blue-Green geändert werden.

| Maßnahme | Beschreibung |
|---|---|
| **Repo-Klassifizierung** | `factory-core` (Poller, Gateway, Infra-Core) vs. `products` (Kunden-Repos). Explizite Liste in Konfiguration. |
| **Forgejo Branch Protection** | `main`-Branch von factory-core Repos: nur via PR, Reviews required, kein Force-Push. |
| **Runner-Scope** | act_runner für Produkt-Repos hat KEINEN Zugriff auf factory-core Repos (separater Token-Scope oder separater Runner). |
| **Poller-Scope** | Poller verarbeitet NUR Produkt-Repos. Factory-Core Änderungen nur via Blue-Green Deployment. |
| **Audit-Trail** | Alle Commits auf factory-core werden geloggt (Autor, Zeitstempel, betroffene Dateien). Anomalie-Erkennung als NICE. |

**Akzeptanztest Paket B:**
1. OpenClaw versucht Push auf factory-core `main` → Rejected (Branch Protection)
2. OpenClaw erstellt PR auf factory-core → PR wird erstellt, aber Merge braucht PO-Approval
3. act_runner (Produkt-Scope) versucht `git clone factory-gateway` → Permission Denied
4. Poller ignoriert factory-core Repos in allen 4 Phasen (dispatch, bridge, openclaw, pr-check)

### Paket C — Backup / Restore vertrauenswürdig (MUSS)

**Ziel:** Backup ist nur so gut wie der letzte erfolgreiche Restore-Test.

| Maßnahme | Beschreibung |
|---|---|
| **Automatisierter Restore-Test** | Wöchentlicher Cron: pg_dump wiederherstellen auf temporärer DB, Forgejo-Repos klonen, Basis-Checks (Row-Count, `git fsck`). Ergebnis per Notification. |
| **Snapshot-Strategie** | UpCloud Server-Snapshots als Ergänzung zu pg_dump. Snapshot vor jedem Blue-Green (bereits in v0.4.0 Backup-Rolle). Retention: 3 Snapshots. |
| **Forgejo Full Dump** | Falls LFS/Attachments genutzt werden: vollständiger Dump (DB + Repos + LFS + Avatare + Attachments). Aktuell nur DB + Git-Repos. |
| **Backup-Monitoring** | Briefkasten-Watchdog prüft: letzter erfolgreicher Backup < 24h, letzter Restore-Test < 7 Tage. Alarm bei Überschreitung. |

**Akzeptanztest Paket C:**
1. Wöchentlicher Restore-Test läuft automatisch → Notification mit Ergebnis (OK/FAIL + Details)
2. Restore-Test schlägt fehl → Alarm-Notification innerhalb 15 Min
3. Backup älter als 24h → Watchdog meldet Alarm
4. Blue-Green erstellt Snapshot vor Switch → Snapshot in UpCloud sichtbar, max 3 Retention

### Paket D — Update / Security Governance (SOLLTE)

**Ziel:** Kontrollierte Updates statt "läuft halt so lange es läuft".

| Maßnahme | Beschreibung |
|---|---|
| **Upgrade-PR-Flow** | Änderungen an `vars.yml` (Version-Pins) erzeugen PR auf infra-core. PO reviewed Changelog + Breaking Changes. Merge triggert Blue-Green auf IAT. |
| **CVE-Scan** | Wöchentlicher `trivy image` Scan aller gepinnten Images. Ergebnis als Issue in infra-core oder Notification. |
| **Secrets-Hygiene** | Vault-Passphrase-Policy (min. 24 Zeichen). Per-Env Deploy Keys (nicht ein Key für alle). Key-Rotation-Plan (halbjährlich). |
| **Break-Glass Access** | Dokumentiertes Runbook: Wer darf wann SSH-Zugang zu PROD nehmen? Audit-Log-Pflicht. Nach Break-Glass: Key rotieren. |

**Akzeptanztest Paket D:**
1. Neue Forgejo-Version in vars.yml → PR auf infra-core mit Changelog-Link
2. `trivy` findet CVE mit Severity HIGH → Issue/Notification innerhalb 24h
3. Deploy Key für DEV kann NICHT auf IAT/PROD zugreifen
4. Break-Glass-Zugang wird genutzt → Log-Eintrag vorhanden, Key-Rotation angestoßen

### Observability Light (MUSS)

> Kein Prometheus/Grafana-Stack — aber ein maschinenlesbarer Status-Endpunkt.

| Komponente | Beschreibung |
|---|---|
| **`/ops/status` API** | JSON-Endpunkt auf Briefkasten-Server. Aggregiert: Poller-Heartbeat, Queue-Länge, letzter Backup-Zeitpunkt, Kill-Switch-State aller Envs, Server-State (UpCloud API). |
| **Strukturiertes Event-Log** | Alle wichtigen Events (Story accepted, Worker started, Backup completed, Kill-Switch toggled) als JSON-Lines in `/var/log/codefabrik-events.jsonl`. Basis für spätere Auswertung. |
| **Notification-Kanäle** | Webhook (ntfy.sh) bleibt primär. Erweiterbar auf Pushover/SMTP. Jede Notification hat Severity (info/warn/critical) und Event-Type. |

### Prioritäts-Klassifizierung

| Priorität | Pakete / Features |
|---|---|
| **MUSS** (Release-Gate) | Paket A (Briefkasten + Watchdog), Paket B (Control Plane Trennung), Paket C (Backup/Restore), Observability Light |
| **SOLLTE** (angestrebt) | Paket D (Update/Security Governance), Release Notes MVP (Generierung aus stories.md + Git-Log, nur technische Perspektive) |
| **NICE** (wenn Zeit) | Voller Dokumenten-Lifecycle, Multi-Produkt parallel, Release Notes mit fachlicher Kundensicht |

### Release Notes — MVP statt Vollausbau

In v0.5.0 als SOLLTE, reduzierter Scope:
- Generierung aus `docs/releases/stories.md` + Git-Log + `_task.md`
- **Nur technische Perspektive** (was wurde geändert, welche Stories, welche Commits)
- Fachliche/Kundensicht erst wenn echte Kunden-Produkte existieren (NICE)
- Braucht Datengrundlage aus v0.4.0 (Story-YAML Felder, stories.md Index)

### DNS-Cutover-Verbesserung

Erweiterung des Blue-Green DNS-Cutover aus v0.4.0:
- **Dual-Stack Verifikation**: Nach DNS-Switch sowohl IPv4 als auch AAAA prüfen (falls konfiguriert)
- **Rollback-Timing**: Messung der tatsächlichen DNS-Propagation-Zeit, nicht nur TTL-basiert warten
- **Cutover-Fenster**: Dokumentiertes Zeitfenster für DNS-Switch (z.B. nur werktags 10-16 Uhr)

### Warum hängt v0.5.0 von v0.4.0 ab?

| v0.5.0 Paket | Abhängigkeit von v0.4.0 |
|---|---|
| A: Briefkasten | Braucht funktionierendes install.yml + nacht-stopp.yml (Motor starten/stoppen per API) |
| B: Control Plane | Braucht Blue-Green (Factory-Core Updates nur via IAT-Test) + Tagging-Konventionen |
| C: Backup/Restore | Braucht funktionierende Backup-Rolle + Verifikation aus v0.4.0 |
| D: Update/Security | Braucht Version-Pinning + Vault-Struktur aus v0.4.0 |
| Observability | Braucht /ready Endpoint + Kill-Switch-Schema als Datenquelle |
| Release Notes MVP | Braucht Story-YAML Felder + stories.md Index |

### v0.5.0 — Ultimativer Akzeptanztest

> Wenn alle diese Punkte erfüllt sind, ist v0.5.0 fertig.

1. **Story-Einreichung → Release ohne manuellen Eingriff**: PO schickt Story via `POST /stories` → Briefkasten queued → Motor startet → Pipeline läuft → PR + Tests → Release auf PROD. PO greift nur für PR-Approval ein.
2. **Ausfall-Erkennung < 5 Min**: PROD /ready antwortet nicht → Watchdog erkennt innerhalb 5 Min → Notification an PO → Optional: Auto-Kill-Switch.
3. **Restore-Test erfolgreich**: Wöchentlicher automatischer Restore-Test bestätigt: DB + Repos sind wiederherstellbar. Letzter Test < 7 Tage.
4. **Self-Modification blockiert**: OpenClaw kann factory-core Repos nicht direkt ändern (nur PR). Runner für Produkte hat keinen Zugriff auf Infra-Repos.
5. **Status jederzeit abfragbar**: `GET /ops/status` zeigt aggregierten Zustand: Queue, Heartbeats, Backup-Alter, Kill-Switch-State, Server-State.
6. **Kontrolliertes Update**: Neue Forgejo-Version → PR auf infra-core → PO-Review → Blue-Green auf IAT → Tests grün → Merge → Blue-Green auf PROD.
