### 3.2 Poller вҖ” `ansible/roles/poller/templates/openclaw-poller.sh.j2`

**Г„nderung:** 3 neue Funktionen + Integration in alle 4 Phasen (77 Zeilen hinzugefГјgt)

**Neue Funktion `retry_with_backoff`:**
```bash
retry_with_backoff() {
  local max_retries=$1
  local description=$2
  shift 2
  local attempt=0
  local wait_seconds=10  # 10s вҶ’ 30s вҶ’ 90s (Г—3 pro Versuch)
  while [ $attempt -lt $max_retries ]; do
    attempt=$((attempt + 1))
    if "$@"; then return 0; fi
    if [ $attempt -lt $max_retries ]; then
      sleep $wait_seconds
      wait_seconds=$((wait_seconds * 3))
    fi
  done
  return 1
}
```

**Neue Funktion `log_poller_event`:**
- Schreibt Events in `poller_events` Tabelle
- Details werden auf 500 Zeichen gekГјrzt
- Fehler werden ignoriert (`|| true`)

**Neue Funktion `record_metric`:**
- Schreibt Timestamps in `pipeline_metrics`
- Idempotent via `INSERT ... ON CONFLICT (object_id) DO UPDATE`

**Integration in die 4 Phasen:**

| Phase | Stelle | Neu hinzugefГјgt |
|-------|--------|-----------------|
| Phase 1 (dispatch) | Nach `update_poller_state "backlog"` | `record_metric dispatched_at` + `product`, `log_poller_event dispatched` |
| Phase 1 (clone) | Process-Repo klonen | `retry_with_backoff 3` statt hartem Fehler, `log_poller_event clone_failed` |
| Phase 2 (bridge) | Nach Branch-Push | `record_metric in_progress_at`, `log_poller_event in_progress` |
| Phase 3 (openclaw) | Claude Code ausfГјhren | `retry_with_backoff 3` statt hartem Fehler, bei ErschГ¶pfung: `record_metric blocked_at + blocked_reason + retry_count` |
| Phase 4 (pr-check) | Nach PR-Erstellung | `record_metric pr_created_at`, `log_poller_event pr_created` |
| Ende | Nach allen Phasen | Retention: `DELETE FROM poller_events WHERE created_at < NOW() - INTERVAL '30 days'` |

**Review-Punkte:**
- `run_openclaw_task()` ist als Funktion definiert, damit `retry_with_backoff` sie aufrufen kann
- Backoff-Faktor Г—3: 10s вҶ’ 30s вҶ’ 90s (Gesamtwartezeit max 130s)
- Retention nur fГјr `poller_events`, nicht `pipeline_metrics` (die bleiben dauerhaft)
- Alle DB-Aufrufe sind fail-soft (`|| true`)

### 4.4 `nightstart.sh.j2` (70 Zeilen) вҖ” Ablauf 08:00 Uhr
1. `docker compose up -d`
2. Warte auf PostgreSQL (max 2 Min / 24Г—5s)
3. Warte auf Forgejo `/api/v1/version` (max 1 Min / 12Г—5s)
4. Kill Switch aufheben: `mode=LIVE`, `jobs_enabled=true`, `traffic_enabled=true`
5. Smoke-Test: Gateway `/ready` muss 200 (max 30s / 6Г—5s)
6. Bei Fehler: `exit 1` (systemd loggt als failed)

### 4.5 systemd Units (je 7-10 Zeilen)
- `codefabrik-nightstop.service` вҖ” Type=oneshot, ExecStart=nightstop.sh
- `codefabrik-nightstop.timer` вҖ” OnCalendar `*-*-* 22:00:00`, Persistent=true
- `codefabrik-nightstart.service` вҖ” Type=oneshot, ExecStart=nightstart.sh
- `codefabrik-nightstart.timer` вҖ” OnCalendar `*-*-* 8:00:00`, Persistent=true

**Review-Punkte:**
- `Persistent=true` sorgt dafГјr, dass verpasste Timer nachgeholt werden
- Postgres wird NICHT gestoppt вҖ” Watchdog und Dispatcher brauchen die DB
- Manuelles Override: `UPDATE runtime_control SET reason='nightstop-disabled'`
- Poller-Wait nutzt PID-Lock (`/tmp/openclaw-poller.lock`) вҖ” konsistent mit Poller-Script
